Step 1: Разметка, CSS-основа и состояния прогресс-бара

На первом этапе собрали статичную часть компонента Step Indicator по мотивам CodePen Jon Kantner. В этом проекте не подключаем SCSS и не ставим зависимости: оригинальная идея переносится в обычные index.html, style.css и позже script.js.

Главная мысль этапа: HTML задает точки и линии, CSS отвечает за геометрию и визуальные состояния, а будущий JS должен только переключать классы step_current и step_done.

HTML-структура

Внутри формы появился блок .steps. Он содержит пять шагов .step, а между соседними шагами стоят линии .step_progress.

Схема DOM 
<div class="steps">
    <div class="step" data-step="0">...</div>
    <div class="step_progress"></div>
    <div class="step" data-step="1">...</div>
    <div class="step_progress"></div>
    ...
</div>
  • data-step начинается с 0, потому что JS удобнее считать шаги как индексы: 0, 1, 2, 3, 4.
  • data-action="prev" и data-action="next" нужны будущей логике: обработчик клика сможет понять действие не по тексту кнопки, а по атрибуту.
  • Классы были названы по-своему: .step, .step_number, .step_name, .step_progress. Это нормально, если CSS и JS используют те же имена.

CSS-переменные и HSL

Переменные компонента положили в .default, а не в :root. Так палитра и настройки анимации относятся только к этому компоненту и не меняют весь хаб.

Component variables 
.default {
    --step-hue: 223;
    --step-primary: hsl(var(--step-hue), 90%, 70%);
    --step-muted: hsl(var(--step-hue), 10%, 40%);
    --step-panel: hsl(var(--step-hue), 10%, 20%);
    --step-duration: 0.3s;
    --step-easing: cubic-bezier(0.65, 0, 0.35, 1);
}

--step-hue

Это не готовый цвет, а число оттенка для hsl(). Поэтому пишем 223, а не #2e3138.

Проценты в HSL

В hsl(223, 90%, 70%) проценты относятся к цвету: 90% - насыщенность, 70% - светлота. Это не ширина и не размер элемента.

Смысл такого подхода: один раз меняем оттенок --step-hue, и вся палитра компонента перестраивается вокруг нового цвета. Если нужен готовый hex-цвет, переменная должна называться по смыслу, например --step-panel: #2e3138, а не --step-hue.

Flex-линия между шагами

Шаги находятся в горизонтальном flex-контейнере. Круги имеют фиксированный размер, а линии между ними должны занимать свободное пространство.

Connector as flex item 
.step_progress {
    flex: 1;
    height: 0.125em;
    background-color: hsl(var(--step-hue), 10%, 40%);
}

Здесь важнее flex: 1, а не width: 100%. Линия является flex-элементом между двумя шагами, поэтому ей нужно не "быть шириной 100% родителя", а "забрать доступное место между соседями".

Абсолютная подпись и неожиданный max-width

Подпись шага вынесена через position: absolute, чтобы она висела под кругом и не раздувала сам flex-ряд.

Step name positioning 
.step {
    position: relative;
}

.step_name {
    position: absolute;
    max-width: unset;
    width: max-content;
    top: 100%;
    left: 50%;
    transform: translateX(-50%);
}
Важный вопрос этапа: почему ширина круга .step_number влияла на ширину подписи .step_name, хотя подпись была absolute?

Ответ оказался в общем файле common.css. Там есть глобальное правило: * { max-width: 100%; }. Абсолютное позиционирование убирает элемент из обычного потока, но не отменяет все ограничения. Поэтому max-width: 100% продолжал ограничивать подпись шириной родителя .step.

А родитель .step получал ширину в основном от круга .step_number. В итоге круг шириной 50px делал родителя примерно 50px, а глобальный max-width: 100% не давал подписи быть шире этого размера. Поэтому добавили max-width: unset и width: max-content.

State-классы: мост между CSS и будущим JS

До написания JS подготовили классы состояний. Это важный порядок: сначала CSS должен знать, как выглядит состояние, а потом JS будет только выбирать, какому шагу это состояние дать.

Step states 
.step_current .step_number,
.step_done .step_number {
    background-color: var(--step-primary);
}

.step_current .step_name,
.step_done .step_name {
    color: hsl(var(--step-hue), 10%, 90%);
    font-weight: bold;
}
  • .step_current - текущий шаг, на котором находится пользователь.
  • .step_done - шаг, который уже пройден.
  • В HTML эти классы не должны оставаться вручную. Они нужны для проверки CSS, а дальше их будет ставить JS.

Псевдоэлемент для заполнения линии

Сама .step_progress остается серым фоном линии. Заполненная синяя часть рисуется поверх через ::before.

Progress fill 
.step_progress {
    position: relative;
    overflow: hidden;
}

.step_progress::before {
    content: "";
    display: block;
    width: 100%;
    height: 100%;
    background-color: var(--step-primary);
    transform: scaleX(0);
    transform-origin: 0 50%;
    transition: transform var(--step-duration) var(--step-easing);
}

scaleX(0) сжимает заполненную линию только по горизонтали. Это точнее, чем scale(0), потому что scale(0) сжимает и ширину, и высоту. Для линии нужен переход scaleX(0) -> scaleX(1).

Соседний селектор

Заполнение линии связано с предыдущим шагом через селектор +. Он выбирает следующий соседний элемент.

Done step fills next connector 
.step_done + .step_progress::before {
    transform: scaleX(1);
}

В нашем HTML после каждого .step сразу идет .step_progress. Поэтому правило читается так: если шаг уже done, заполни линию, которая стоит сразу после него.

Что осталось на потом

cubic-bezier(0.65, 0, 0.35, 1) пока зафиксирован как настройка скорости перехода. Подробный разбор лучше делать на следующем CSS-динамическом этапе, когда будет видно движение: это кривая, которая управляет тем, как анимация разгоняется и тормозит между началом и концом.

appearance: none у кнопки убирает системный стиль браузера, чтобы наша кнопка выглядела одинаковее в разных браузерах и ОС. Это не логика компонента, а нормализация внешнего вида.

Итог Step 1

  1. Собрана HTML-структура: шаги, линии между шагами и кнопки управления.
  2. data-step и data-action подготовлены под будущую JS-логику.
  3. CSS-переменные вынесены в область компонента .default.
  4. Разобрано, что --step-hue - это число оттенка для HSL, а проценты в HSL относятся к насыщенности и светлоте.
  5. Линия между шагами сделана flex-элементом через flex: 1.
  6. Подпись шага позиционируется абсолютно относительно .step.
  7. Проблема с шириной подписи была из-за глобального max-width: 100% из common.css.
  8. Подготовлены состояния .step_current и .step_done.
  9. Заполнение линии сделано через .step_progress::before, scaleX() и соседний селектор +.
  10. Следующий шаг: JS-логика, которая будет хранить номер текущего шага, переключать классы и включать/выключать кнопки.

Step 2: JS-логика переключения шагов

На втором этапе компонент стал интерактивным: кнопки Previous и Next меняют индекс текущего шага, а JS заново расставляет CSS-классы step_done и step_current.

Главная мысль JS-этапа: JavaScript не рисует прогресс-бар напрямую. Он только хранит число currentStep и переключает классы. Вся визуальная динамика находится в CSS.

Поиск элементов

Сначала нашли контейнер, шаги и кнопки. Для кнопок использовали не класс .btn, а атрибут data-action, потому что кнопок две и у каждой своя команда.

DOM references 
const stepsContainer = document.querySelector(".steps");
const stepItems = document.querySelectorAll(".step");
const prevBtn = document.querySelector('[data-action="prev"]');
const nextBtn = document.querySelector('[data-action="next"]');

Ошибка, которая возникла по пути: попытка искать кнопки через document.querySelector([".btn"]). Метод querySelector принимает строку CSS-селектора, а не массив. Кроме того, селектор .btn нашел бы только первую кнопку.

Защитная проверка

Если какого-то важного элемента нет, скрипт должен остановиться и не падать дальше.

Guard clause 
if ((!stepsContainer || !stepItems.length) || (!prevBtn || !nextBtn)) {
    return;
}

Эту проверку можно записать проще, смысл будет тот же:

Readable guard 
if (!stepsContainer || !stepItems.length || !prevBtn || !nextBtn) {
    return;
}

currentStep и maxStep

currentStep хранит текущий индекс шага. Индекс начинается с нуля, поэтому первый кружок на экране имеет индекс 0, а пятый - индекс 4.

State values 
let currentStep = 0;
const currentClass = "step_current";
const doneClass = "step_done";
const maxStep = stepItems.length - 1;

Для человека

На экране шаги выглядят как 1, 2, 3, 4, 5.

Для JavaScript

В коде индексы идут как 0, 1, 2, 3, 4. Поэтому последний индекс равен length - 1.

Это объясняет вопрос, почему при currentStep = 4 прогресс уже почти полностью заполнен: индекс 4 - это пятый, последний шаг.

renderStep: одна функция рисует все состояния

Функция renderStep проходит по всем шагам и для каждого решает: он уже пройденный, текущий или будущий.

Render logic 
function renderStep(targetStep) {
    stepItems.forEach((step, index) => {
        step.classList.remove(currentClass, doneClass);

        if (targetStep === index) {
            step.classList.add(currentClass);
        } else if (targetStep > index) {
            step.classList.add(doneClass);
        }
    });

    updateButtons();
}

Здесь была важная ошибка с classList.remove. Запись currentClass || doneClass не передает два класса. Оператор || выбирает одно значение. То же самое с &&: это не "и два аргумента", а логический оператор.

Remove two classes 
// Неправильно: удалится только один выбранный оператором класс
step.classList.remove(currentClass || doneClass);

// Правильно: remove принимает несколько аргументов через запятую
step.classList.remove(currentClass, doneClass);

updateButtons: disabled как результат проверки

Сначала было желание писать условия только на выключение кнопок. Но такая логика работает в одну сторону: кнопка выключилась и потом сама не включится обратно.

Button state 
function updateButtons() {
    prevBtn.disabled = currentStep <= 0;
    nextBtn.disabled = currentStep >= maxStep;
}

Здесь справа от = стоит проверка, которая возвращает true или false. Каждый раз состояние кнопок пересчитывается заново.

Еще одна ошибка этапа: desable вместо disabled. JavaScript не ругается, потому что можно создать новое случайное свойство на объекте. Но браузерная кнопка реагирует именно на disabled.

Клики Previous и Next

Обработчики кнопок не трогают классы напрямую. Они только меняют число currentStep, а потом вызывают renderStep(currentStep).

Click handlers 
prevBtn.addEventListener("click", () => {
    if (currentStep > 0) {
        currentStep--;
        renderStep(currentStep);
    }
});

nextBtn.addEventListener("click", () => {
    if (currentStep < maxStep) {
        currentStep++;
        renderStep(currentStep);
    }
});

Для Previous важна левая граница: currentStep > 0. Для Next важна правая граница: currentStep < maxStep. Условие currentStep <= 0 || currentStep < maxStep работало, но было логически шумным: Next не должен думать про левую границу.

Итог Step 2

  1. Кнопки нашли через data-action, потому что это смысловой крючок команды.
  2. currentStep хранит индекс текущего шага.
  3. maxStep равен последнему индексу, поэтому это length - 1.
  4. renderStep сначала очищает классы, потом выставляет done/current.
  5. classList.remove принимает несколько классов через запятую.
  6. disabled = условие проще и надежнее, чем отдельные if только на выключение.
  7. Клики меняют только число и запускают перерисовку; визуальная работа остается в CSS.

CSS Deep Dive: почему прогресс-бар работает без сложного JS

Этот раздел фиксирует самый сложный кусок проекта. Разметка и JS получились понятнее, а стили оказались тяжелыми, потому что в одном компоненте встретились сразу несколько CSS-механик: mobile-first, container query, flex, absolute-позиционирование, псевдоэлемент, transform по разным осям и задержка перехода.

Главная мысль CSS-части: JS только ставит классы step_done и step_current. Все ощущение "прогресс доехал до следующего шага" создают CSS-селекторы, ::before, transform и transition-delay.

1. Один повторяющийся HTML-паттерн

Вся CSS-магия держится на порядке элементов в HTML. После каждого шага идет линия. Именно этот порядок позволяет использовать соседний селектор +.

Step + connector pattern 
<div class="step" data-step="0">
    <div class="step_number">1</div>
    <div class="step_name">About You</div>
</div>

<div class="step_progress"></div>

CSS потом читает это так: если шаг получил класс step_done, то линия сразу после него должна заполниться.

Adjacent selector 
.step_done + .step_progress::before {
    transform: scaleY(1);
}

Плюс + выбирает не всех соседей, а только следующий элемент сразу после текущего. Поэтому линия заполняется не где попало, а ровно после завершенного шага.

2. Mobile-first: почему "все наоборот"

Сначала мы сделали горизонтальную desktop-версию. Потом выяснилось, что оригинал построен наоборот: базовые стили описывают мобильную вертикальную версию, а широкая версия включается отдельным блоком.

Desktop-first

Сначала пишем широкую версию, потом ломаем ее медиазапросами под мобилку. Для этого компонента так сложнее, потому что ось линии меняется с X на Y.

Mobile-first

Сначала пишем простую вертикальную версию. Потом в @container переопределяем только отличия desktop-версии.

В мобильной версии шаг - это обычная строка: круг слева, текст справа. Поэтому .step_name не должен быть absolute.

Mobile idea: обычный поток

1About You
2About Book

Desktop idea: линия по X

1 2 3

3. Контейнер формы управляет раскладкой

В оригинале используется container query. Это похоже на media query, но условие смотрит не на ширину всего окна браузера, а на ширину конкретного контейнера.

Enable container query 
.default form {
    container: form / inline-size;
    width: 100%;
    max-width: 100%;
    margin: 0 auto;
}

container: form / inline-size говорит браузеру: эта форма теперь контейнер с именем form, и ее можно проверять по inline-size, то есть по ширине в обычном горизонтальном письме.

Desktop switch 
@container form (min-width: 30em) {
    .steps {
        flex-direction: row;
        align-items: center;
        padding: 70px;
    }
}

min-width: 30em значит: когда форма стала достаточно широкой, включи desktop-раскладку. em здесь зависит от размера шрифта: если базовый шрифт 16px, то 30em примерно 480px.

4. .steps: внешний режим списка

Базовый .steps теперь отвечает за мобильную вертикальную версию.

Mobile .steps 
.steps {
    display: flex;
    flex-direction: column;
    justify-content: center;
    padding: 1.5rem;
    background-color: var(--step-panel);
    border-radius: 12px;
}
  • display: flex делает детей управляемыми flex-элементами.
  • flex-direction: column ставит шаги и линии сверху вниз.
  • padding: 1.5rem дает внутренний воздух без огромной фиксированной высоты на мобилке.

В desktop-режиме меняется только направление и выравнивание:

Desktop .steps override 
@container form (min-width: 30em) {
    .steps {
        flex-direction: row;
        align-items: center;
        padding: 70px;
    }
}

Это не два разных компонента. Это один и тот же HTML, но ось раскладки меняется: mobile - ось Y, desktop - ось X.

5. .step: строка на мобилке, точка на desktop

На мобилке .step должен быть обычной строкой. Круг и текст стоят рядом, поэтому absolute-позиционирование не нужно.

Step row 
.step {
    display: flex;
    align-items: center;
    flex-shrink: 0;
    position: relative;
    z-index: 1;
}

flex-shrink: 0 запрещает шагу сжиматься, когда рядом появляются линии. z-index: 1 держит шаг выше линии, если они визуально пересекаются.

6. .step_number: круг и анимация состояния

Круг - это обычный flex-блок с одинаковыми width и height. На мобилке у него есть правый отступ до текста.

Number circle 
.step_number {
    display: flex;
    align-items: center;
    justify-content: center;
    border-radius: 50%;
    width: 50px;
    height: 50px;
    margin-inline-end: 0.5em;
    transition:
        background-color var(--step-duration) var(--step-easing),
        box-shadow var(--step-duration) var(--step-easing),
        color var(--step-duration) var(--step-easing);
}

margin-inline-end - это логический аналог отступа справа. В обычном направлении текста он работает как margin-right, но лучше подходит для интерфейсов, где направление письма теоретически может быть другим.

transition стоит на базовом классе, а не только на .step_current. Поэтому изменение плавное и при добавлении класса, и при удалении.

7. .step_name: почему absolute только на desktop

На мобилке подпись - обычный текст рядом с кругом. Поэтому она стоит в потоке и участвует в ширине строки.

Mobile step name 
.step_name {
    padding: 10px;
    max-width: unset;
    width: max-content;
    color: hsl(var(--step-hue), 10%, 50%);
    font-size: 20px;
    line-height: 1.2;
    transition:
        color var(--step-duration) var(--step-easing),
        font-weight var(--step-duration) var(--step-easing);
}

В desktop-режиме подпись нужно повесить под кругом, поэтому там возвращается absolute:

Desktop step name 
@container form (min-width: 30em) {
    .step_name {
        position: absolute;
        top: 100%;
        left: 50%;
        transform: translateX(-50%);
        padding: 10px;
    }
}

top: 100% ставит верх подписи сразу под высотой родителя .step. left: 50% ставит левый край в центр родителя. translateX(-50%) сдвигает подпись назад на половину ее собственной ширины, поэтому она центрируется под кругом.

8. .step_progress: линия как отдельный элемент

Линия между шагами - это не border у шага. Это отдельный элемент .step_progress. Он нужен, чтобы линия могла быть серым фоном, а поверх нее могло ехать синее заполнение.

Mobile vertical connector 
.step_progress {
    position: relative;
    overflow: hidden;
    width: 0.125em;
    height: 1.25em;
    margin-inline-start: 25px;
    background-color: hsl(var(--step-hue), 10%, 40%);
    transform: translateX(-50%);
}
  • width: 0.125em - тонкая вертикальная линия.
  • height: 1.25em - длина линии между двумя шагами.
  • margin-inline-start: 25px - сдвиг к центру круга 50px.
  • translateX(-50%) центрирует саму тонкую линию относительно этой точки.
  • overflow: hidden прячет ту часть ::before, которая сжата transform-ом.

На desktop линия становится горизонтальной и занимает свободное место между кругами:

Desktop horizontal connector 
@container form (min-width: 30em) {
    .step_progress {
        flex: 1;
        width: auto;
        height: 0.125em;
        margin-inline-start: 0;
        transform: none;
    }
}

flex: 1 здесь означает: линия забирает свободное место между шагами. Это причина, почему на широком экране линии растягиваются ровно между кружками.

9. ::before: синее заполнение поверх серой линии

Серый фон находится на самой .step_progress. Синяя заполняющая часть находится на псевдоэлементе .step_progress::before.

Mobile fill by Y axis 
.step_progress::before {
    transform: scaleY(0);
    transform-origin: 50% 0;
    content: "";
    display: block;
    width: 100%;
    height: 100%;
    background-color: var(--step-primary);
    transition: transform var(--step-duration) var(--step-easing);
}

В мобильной версии линия вертикальная, поэтому используется scaleY. scaleY(0) - высота заполнения сжата до нуля. scaleY(1) - заполнение на всю высоту.

transform-origin: 50% 0 задает точку роста: по X центр, по Y верх. Значит синяя линия растет сверху вниз.

Desktop fill by X axis 
@container form (min-width: 30em) {
    .step_progress::before {
        transform: scaleX(0);
        transform-origin: 0 50%;
    }
}

В desktop-версии линия горизонтальная, поэтому используется scaleX. transform-origin: 0 50% значит: расти от левого края к правому.

10. step_done запускает заполнение следующей линии

Когда JS ставит шагу класс step_done, CSS находит следующую после него линию и раскрывает ее ::before.

Mobile done connector 
.step_done + .step_progress::before {
    transform: scaleY(1);
}
Desktop done connector 
@container form (min-width: 30em) {
    .step_done + .step_progress::before {
        transform: scaleX(1);
    }
}

Здесь находится основной "секрет" компонента. JS не анимирует ширину линии. JS только говорит: этот шаг уже done. CSS сам знает, что линия после done-шага должна заполниться.

11. Почему current подсвечивается после линии

Проблема была такая: при клике новый кружок сразу становился активным, а линия еще только ехала до него. Визуально получалось нелогично: будто пользователь уже перешел на следующий шаг быстрее, чем прогресс дошел до него.

Решение: линия стартует сразу, а подсветка current получает задержку на длительность этой линии.

Delayed current number 
.step_current .step_number {
    transition-delay: var(--step-duration);
    background-color: var(--step-primary);
    box-shadow: 0 0 0 4px hsl(223deg 90% 70% / 75%);
    color: hsl(var(--step-hue), 10%, 90%);
}

--step-duration равен 0.3s. Линия тоже анимируется 0.3s. Поэтому кружок ждет столько же, сколько линия заполняется.

Важно: задержка работает заметно только потому, что у базового .step_number есть transition. Если transition нет, браузер просто мгновенно применит новые стили.

12. cubic-bezier простыми словами

cubic-bezier(0.65, 0, 0.35, 1) управляет не направлением анимации, а ее скоростью во времени. Это не "куда едет линия", а "как она разгоняется и тормозит".

Transition variables 
--step-duration: 0.3s;
--step-easing: cubic-bezier(0.65, 0, 0.35, 1);
  • duration - сколько длится переход.
  • easing - как меняется скорость внутри этого времени.
  • linear ехал бы одинаково ровно.
  • cubic-bezier дает более живое движение с разгоном и торможением.

13. Полная связка HTML, CSS и JS

На одном переходе Next вся цепочка выглядит так.

  1. Пользователь нажимает Next.
  2. JS проверяет currentStep < maxStep.
  3. JS увеличивает currentStep.
  4. renderStep очищает классы у всех шагов.
  5. Шаги с индексом меньше текущего получают step_done.
  6. Текущий шаг получает step_current.
  7. CSS-селектор .step_done + .step_progress::before запускает заполнение линии.
  8. .step_current .step_number подсвечивается с задержкой.
  9. updateButtons пересчитывает disabled у кнопок.
The whole idea in short 
// JS
if (targetStep > index) {
    step.classList.add("step_done");
}

// CSS
.step_done + .step_progress::before {
    transform: scaleY(1);
}

Это и есть причина, почему прогресс-бар начал работать "правильно" без усложнения JS. Правильная динамика была заложена в CSS-структуре: класс на шаге управляет линией, которая стоит после него.

14. Что писать, если собирать компонент заново

  1. Сначала HTML: step -> progress -> step -> progress.
  2. Потом CSS mobile-first: вертикальная колонка, шаг строкой, линия по Y.
  3. Потом state-классы: step_done, step_current.
  4. Потом ::before на линии и заполнение через scaleY.
  5. Потом @container и desktop-переопределения: row, линия по X, подпись absolute.
  6. Потом delay для current, чтобы визуальный порядок стал логичным.
  7. И только после этого JS, который переключает классы.

Итог CSS Deep Dive

  1. Mobile-first здесь нужен не для моды, а потому что вертикальная версия проще как базовая.
  2. @container переключает компонент по ширине формы, а не всего окна.
  3. Один и тот же HTML работает в двух раскладках: mobile по Y, desktop по X.
  4. .step_progress - серый фон линии, ::before - синее заполнение.
  5. scaleY нужен для вертикальной линии, scaleX для горизонтальной.
  6. transform-origin задает точку, откуда растет заполнение.
  7. .step_done + .step_progress::before связывает завершенный шаг со следующей линией.
  8. transition-delay делает порядок понятным: сначала линия, потом активный кружок.
  9. JS остался простым именно потому, что CSS взял на себя визуальную механику.
