Step 1: Каркас, доска и создание колонки

На первом шаге собрали основу task manager: верхняя форма добавляет колонки, внутри доски есть горизонтальная область для колонок, а каждая колонка содержит форму для будущего добавления задач и отдельный список задач.

Главная идея этапа: доска - это не просто список карточек. У нее есть уровни ответственности: viewport для скролла, ряд колонок, отдельная колонка, форма задачи и список задач.

Структура интерфейса

Разметка строится вокруг нескольких смысловых уровней. Верхняя форма находится отдельно от scroll-области, чтобы не уезжать вместе с колонками.

DOM structure 
<section class="task-manager">
    <form class="add-column-form">...</form>

    <div class="task-board">
        <ul class="task-board__columns" id="columns">
            <li class="task-column">
                <header class="task-column__header">...</header>
                <form class="add-task-form">...</form>
                <ul class="task-list">...</ul>
            </li>
        </ul>
    </div>
</section>

БЭМ-нейминг

В проекте начали использовать БЭМ. Двойное подчеркивание показывает, что элемент является частью блока. Например, task-column__title - это title внутри блока task-column.

BEM pattern 
.task-column
.task-column__header
.task-column__title

.add-task-form
.add-task-form__input
.add-task-form__button

.task
.task__text
.task__delete-button

Важное правило: если есть класс вида block__element, должен быть понятен блок block. Поэтому add-task-form__input логичнее, чем add-task__input, если блока add-task в разметке нет.

Горизонтальная доска

Для колонок нужна отдельная область прокрутки. Важно не расширять всю страницу, а скроллить только внутреннюю доску.

Board viewport 
.task-board {
    width: 100%;
    overflow-x: auto;
    overflow-y: hidden;
}
  • overflow-x: auto включает горизонтальный скролл, когда колонок больше ширины контейнера.
  • overflow-y: hidden не дает board-обертке создавать лишний вертикальный скролл.
  • overflow-x: hidden здесь не подходит: он просто обрежет лишние колонки.

Ряд колонок

Сами колонки лежат внутри ul.task-board__columns. Для такого ряда удобнее flex, потому что новые колонки просто добавляются в конец линии.

Columns row 
.task-board__columns {
    display: flex;
    align-items: flex-start;
    gap: 16px;
    width: max-content;
}

width: max-content говорит ряду колонок: будь такой ширины, сколько нужно твоему содержимому. Если колонок много, ряд становится шире viewport, а .task-board дает горизонтальный скролл.

Ширина колонки

Колонка должна быть стабильной ширины. Для этого используется flex-сокращение:

Stable column width 
.task-column {
    flex: 0 0 280px;
}
  • Первый 0 - flex-grow: колонка не растет.
  • Второй 0 - flex-shrink: колонка не сжимается.
  • 280px - базовая ширина колонки.

Поэтому width: 100% на самой колонке здесь не нужен: он конфликтует по смыслу с задачей держать фиксированную ширину внутри flex-ряда.

Стандартный padding у ul

У ul в браузере по умолчанию есть левый отступ около 40px. Обнулить только list-style недостаточно: маркеры исчезнут, но padding останется.

List reset 
.task-board__columns,
.task-list {
    margin: 0;
    padding: 0;
    list-style: none;
}

Потом для .task-list можно отдельно вернуть нужный внутренний padding уже осознанно, например для отступов карточек внутри колонки.

Гибкий input и кнопка без переноса

У кнопки добавления колонки текст переносился на две строки. Это не проблема ширины формы, а поведение текста внутри кнопки.

No wrapping button text 
.add-column-form__button {
    white-space: nowrap;
}

white-space: nowrap запрещает перенос текста внутри кнопки. Input при этом может оставаться гибким и занимать свободное место.

DOMContentLoaded

JS запускается после события DOMContentLoaded, чтобы элементы из HTML уже существовали и их можно было найти через querySelector.

DOM ready 
document.addEventListener("DOMContentLoaded", () => {
    // Здесь уже можно искать элементы.
});

Проверка элементов

После поиска элементов идет проверка. Если какой-то важный элемент не найден, код останавливается через return.

Guard check 
if (!addColumnForm || !addColumnInput || !columnsItem) {
    console.log("Task manager elements not found");
    return;
}

return работает здесь, потому что код находится внутри функции обработчика DOMContentLoaded. На верхнем уровне файла return вызвал бы ошибку.

submit и preventDefault

Форма добавления колонки обрабатывается через событие submit. Это покрывает сразу два сценария: клик по кнопке submit и Enter внутри input.

Form submit 
addColumnForm.addEventListener("submit", (e) => {
    e.preventDefault();

    const columnTitle = addColumnInput.value.trim();
});

У формы есть стандартное поведение браузера: отправить форму и перезагрузить страницу. e.preventDefault() отменяет это стандартное действие. После этого форма остается на странице, а логикой управляет JS.

createElement и classList.add

Новая колонка создается через DOM API, а не через innerHTML. Это хорошая практика, потому что название колонки вводит пользователь.

Create DOM element 
const newColumn = document.createElement("li");
newColumn.classList.add("task-column");

document.createElement("li") создает новый DOM-элемент в памяти. Он еще не находится на странице. classList.add(...) добавляет CSS-класс этому элементу.

Свойства DOM-элементов

У созданного input можно настраивать свойства напрямую. Это не CSS-свойства, а свойства DOM-объекта, которые соответствуют HTML-атрибутам.

Input properties 
newColumnInput.type = "text";
newColumnInput.placeholder = "New task...";
newColumnInput.maxLength = 40;
  • type соответствует HTML-атрибуту type.
  • placeholder соответствует HTML-атрибуту placeholder.
  • maxLength соответствует HTML-атрибуту maxlength, но в JS пишется camelCase.

То есть эта JS-запись:

JS properties 
newColumnInput.type = "text";
newColumnInput.placeholder = "New task...";
newColumnInput.maxLength = 40;

примерно соответствует такой HTML-разметке:

HTML attributes 
<input type="text" placeholder="New task..." maxlength="40">

textContent

Пользовательский текст записывается через textContent. Это важно: textContent вставляет именно текст, а не HTML.

Safe text insertion 
newColumnTitle.textContent = columnTitle;

Если использовать innerHTML с пользовательским вводом, браузер будет воспринимать строку как HTML-разметку. Для портфолио и хорошей практики пользовательский текст лучше вставлять через textContent.

append

createElement создает отдельные элементы, но они еще не связаны между собой. append собирает эти элементы в DOM-дерево.

Append tree 
newColumnHeader.append(newColumnTitle);
newColumnForm.append(newColumnInput, newColumnButton);
newColumn.append(newColumnHeader, newColumnForm, taskList);
columnsItem.append(newColumn);

Сначала собираются маленькие части: заголовок получает title, форма получает input и button. Потом эти части добавляются внутрь колонки. Последний columnsItem.append(newColumn) вставляет готовую колонку в настоящий ul#columns на странице.

Mental model 
li.task-column
├─ header.task-column__header
│  └─ h2.task-column__title
├─ form.add-task-form
│  ├─ input.add-task-form__input
│  └─ button.add-task-form__button
└─ ul.task-list

Очистка input

После успешного создания колонки верхний input очищается:

Clear input 
addColumnInput.value = "";

Это setter-запись: свойству value input-элемента задается пустая строка. Пользователь видит, что форма готова принять следующее название колонки.

setAttribute и aria-label

Если кнопка создается через JS, ее HTML-атрибуты тоже можно задать через JS. Для кнопок с символом × это важно: визуально понятно, что кнопка удаляет, но скринридеру нужен текстовый смысл действия.

Set accessibility label 
newColumnDeleteButton.setAttribute("aria-label", "Delete column");
newTaskDeleteButton.setAttribute("aria-label", "Delete task");

setAttribute(name, value) ставит HTML-атрибут на DOM-элемент. Эта JS-запись соответствует такой разметке:

HTML result 
<button aria-label="Delete column">×</button>
<button aria-label="Delete task">×</button>

e.target.closest()

e.target - это конкретный элемент, с которого началось событие. Например, при клике по кнопке удаления target обычно будет сама кнопка. При submit формы target обычно будет форма.

Event target 
columnsItem.addEventListener("click", (e) => {
    console.log(e.target);
});

Но в динамическом интерфейсе нам обычно нужен не просто элемент, по которому кликнули, а ближайший смысловой контейнер: кнопка удаления, задача, колонка или форма. Для этого используется closest().

Closest button 
const deleteButton = e.target.closest(".task__delete-button");

Эта строка читается так: "начни с элемента, где произошло событие, и поднимайся вверх по DOM, пока не найдешь ближайший элемент с классом task__delete-button". Если такой элемент не найден, вернется null.

Важное уточнение: closest() ищет только вверх по DOM-дереву: сначала проверяет сам элемент, потом его родителя, потом родителя родителя и так далее. Он не ищет соседей и не ищет дочерние элементы.
Direction of closest 
button.task__delete-button
└─ closest(".task")
   └─ li.task

Для поиска вниз внутри уже найденного контейнера используется querySelector().

closest up, querySelector down 
const currentTask = deleteButton.closest(".task"); // вверх к li.task
const taskText = currentTask.querySelector(".task__text"); // вниз внутрь li.task
Guard after closest 
if (!deleteButton) {
    return;
}

Такая проверка нужна при делегировании событий. Обработчик висит на большом родителе columnsItem, поэтому клик может произойти где угодно внутри доски: по input, по заголовку, по карточке, по кнопке. Если клик был не по кнопке удаления, обработчик просто выходит через return.

Несколько действий в одном click-обработчике

Удаление задачи и удаление колонки можно держать в одном делегированном click-обработчике на columnsItem. Но проверки должны идти по очереди: сначала один сценарий, затем другой.

Delegated click scenarios 
columnsItem.addEventListener("click", (e) => {
    const taskDeleteButton = e.target.closest(".task__delete-button");

    if (taskDeleteButton) {
        const currentTask = taskDeleteButton.closest(".task");
        currentTask.remove();
        return;
    }

    const columnDeleteButton = e.target.closest(".task-column__delete-button");

    if (columnDeleteButton) {
        const currentColumn = columnDeleteButton.closest(".task-column");
        currentColumn.remove();
    }
});

Общая проверка вроде if (!taskDeleteButton || !columnDeleteButton) здесь не подходит. При клике по кнопке задачи кнопка колонки будет null, а при клике по кнопке колонки кнопка задачи будет null. Поэтому это не проверка "оба должны существовать", а последовательный выбор сценария.

Маршрут до текущей задачи

После того как кнопка удаления найдена, нужно удалить не кнопку, а всю задачу li.task. Поэтому от кнопки поднимаемся к ближайшей задаче.

Delete route 
const currentTask = deleteButton.closest(".task");
currentTask.remove();

Модель маршрута:

Click path 
click
└─ e.target
   └─ closest(".task__delete-button")
      └─ closest(".task")
         └─ remove()

closest при добавлении задачи

Та же идея используется при добавлении задачи. Нужно понять, из какой колонки пришел submit. Мы начинаем с формы задачи и поднимаемся к ближайшей колонке.

Current column route 
const taskForm = e.target.closest(".add-task-form");
const currentColumn = taskForm.closest(".task-column");
const taskList = currentColumn.querySelector(".task-list");

Так задача добавляется не в первую колонку и не во все колонки, а именно в список той колонки, где пользователь нажал Add.

Здесь маршрут смешанный: сначала два раза идем вверх через closest(), чтобы понять контекст, а потом идем вниз через querySelector(), чтобы найти нужный список внутри этой колонки.

closest() - это способ найти контекст действия. Кнопка говорит "удали мою задачу", форма говорит "добавь задачу в мою колонку".

typeof и проверка библиотеки

Для Dragula добавилась проверка: загрузилась ли библиотека до нашего script.js. Библиотека приходит из CDN и создает глобальную функцию window.dragula.

Library guard 
if (typeof window.dragula === "undefined") {
    console.log("Dragula is not loaded");
    return;
}

Важный момент: оператор typeof возвращает строку. Поэтому результат сравнивается со строкой "undefined", а не с обычным значением undefined.

typeof returns string 
typeof 123          // "number"
typeof "hello"      // "string"
typeof true         // "boolean"
typeof undefined    // "undefined"
typeof function(){} // "function"

Такая проверка безопасна для глобальных библиотек: если Dragula не загрузилась, код не упадет на вызове dragula(...), а спокойно остановится с понятным сообщением в консоли.

Итог Step 1

  1. Разметка разделена на верхнюю форму, viewport доски, ряд колонок, колонку, форму задачи и список задач.
  2. Для горизонтальной доски используются overflow-x: auto, display: flex и width: max-content.
  3. Колонка держит стабильную ширину через flex: 0 0 280px.
  4. Стандартный левый padding у ul нужно обнулять отдельно.
  5. submit формы удобнее, чем отдельные обработчики click и Enter.
  6. preventDefault() отменяет стандартную отправку формы.
  7. createElement создает DOM-элемент в памяти.
  8. classList.add добавляет CSS-класс элементу.
  9. textContent безопасно вставляет пользовательский текст.
  10. append собирает элементы в дерево и вставляет готовую колонку в страницу.
  11. setAttribute задает HTML-атрибуты созданным через JS элементам.
  12. e.target.closest(...) помогает найти контекст события: форму, колонку, кнопку или задачу.
  13. typeof возвращает строку, поэтому проверка пишется как typeof value === "undefined".

Step 2: Состояние доски и localStorage

На этом этапе доска получила постоянное состояние. Колонки, задачи и порядок задач сохраняются после каждого изменения, а после перезагрузки страницы DOM восстанавливается из сохраненных данных.

Главная модель этапа: DOM → обычные JS-данные → JSON-строка → localStorage → JSON-строка → обычные JS-данные → DOM.

Почему нельзя сохранить DOM напрямую

DOM-элементы являются объектами браузера со свойствами, связями и методами. localStorage хранит только строки, поэтому сначала доску нужно описать простой структурой данных.

Board state 
[
    {
        title: "To Do",
        tasks: ["Read documentation", "Make layout"]
    },
    {
        title: "Done",
        tasks: ["Create first column"]
    }
]

Один объект описывает одну колонку. Свойство title содержит строку заголовка, а tasks - массив строк задач. Такая структура не зависит от HTML и может быть преобразована в JSON.

DOM → массив объектов

Функция getBoardState() читает актуальный DOM. Внешний map() превращает каждую колонку в объект, а внутренний map() превращает элементы задач в строки.

getBoardState 
function getBoardState() {
    const columns = Array.from(
        columnsItem.querySelectorAll(".task-column")
    );

    return columns.map((column) => {
        const title = column
            .querySelector(".task-column__title")
            .textContent;

        const tasks = Array.from(
            column.querySelectorAll(".task__text")
        ).map((taskText) => {
            return taskText.textContent;
        });

        return {
            title: title,
            tasks: tasks
        };
    });
}

querySelectorAll() возвращает NodeList. Через Array.from() коллекция превращается в обычный массив, после чего доступен привычный map().

В объекте title: title слева находится имя свойства, справа - значение локальной переменной. То же относится к tasks: tasks.

JSON.stringify и сохранение

Массив нельзя положить в localStorage как массив. Сначала JSON.stringify() сериализует его в строку.

saveBoard 
const STORAGE_KEY = "task-manager-board";

function saveBoard() {
    const boardState = getBoardState();
    const boardStateJSON = JSON.stringify(boardState);

    localStorage.setItem(STORAGE_KEY, boardStateJSON);
}

Ключ - это имя записи. По тому же ключу приложение позднее получает данные. В панели Application значение выглядит как массив, потому что DevTools умеет показывать разобранный JSON, но фактически в хранилище находится строка.

Сохраняем после изменения DOM

Порядок операций принципиален:

Mutation order 
изменить DOM;
saveBoard();

Если вызвать saveBoard() до append() или remove(), функция прочитает старое состояние. Сохранение выполняется после:

  1. создания колонки;
  2. создания задачи;
  3. удаления задачи;
  4. удаления колонки;
  5. завершенного drag and drop.

Событие Dragula drop

Перемещением управляет библиотека, поэтому новый порядок сохраняется через ее событие drop. К моменту вызова обработчика Dragula уже переставила задачу в DOM, и getBoardState() прочитает новый порядок.

Save after drop 
drake.on("drop", function () {
    saveBoard();
});

Чтение: JSON.parse и стабильный тип результата

localStorage.getItem() возвращает строку, если ключ найден, и null, если ключа нет. Это контракт метода: отсутствующая запись здесь не равна undefined.

loadBoard 
function loadBoard() {
    const savedBoardJSON = localStorage.getItem(STORAGE_KEY);

    if (savedBoardJSON === null) {
        return [];
    }

    return JSON.parse(savedBoardJSON);
}

При отсутствии записи функция возвращает пустой массив, а не обычный return. Обычный return вернул бы undefined, после чего вызов forEach() завершился бы ошибкой.

Ключ отсутствует

getItem(STORAGE_KEY) // null
loadBoard()          // []

Сохранена пустая доска

getItem(STORAGE_KEY) // "[]"
loadBoard()          // []

Для приложения результат одинаковый - пустой массив. Но состояние хранилища разное: в первом случае ключа нет, во втором ключ существует и содержит строку "[]".

Массив объектов → DOM

renderBoard(boardState) выполняет обратное преобразование. Сначала контейнер очищается через replaceChildren(), затем внешний forEach() создает колонки, а вложенный forEach() - задачи текущей колонки.

Nested data traversal 
function renderBoard(boardState) {
    columnsItem.replaceChildren();

    boardState.forEach((columnData) => {
        // columnData - объект одной колонки
        newColumnTitle.textContent = columnData.title;

        columnData.tasks.forEach((taskText) => {
            // taskText - одна строка из массива tasks
            newTaskTitle.textContent = taskText;
        });
    });
}

columnData.title является одной строкой. columnData.tasks является массивом, поэтому его нужно перебрать. На каждой итерации внутреннего цикла taskText содержит текст одной задачи.

В renderBoard() нет события e, формы, введенного значения или taskInput. Источник данных здесь не пользовательский submit, а параметр boardState.

Порядок инициализации Dragula

Каждый восстановленный taskList регистрируется через drake.containers.push(taskList). Поэтому вызов renderBoard() выполняется после создания drake.

Initialization order 
const drake = window.dragula(taskLists);

drake.on("drop", function () {
    saveBoard();
});

const savedBoard = loadBoard();
renderBoard(savedBoard);

Функция renderBoard может быть объявлена выше drake, потому что выполняется она позже. В момент фактического вызова переменная уже инициализирована.

Область видимости drake

Переменная drake создана внутри обработчика DOMContentLoaded. Поэтому код приложения внутри этого обработчика видит переменную, а глобальная консоль браузера - нет.

Local scope 
document.addEventListener("DOMContentLoaded", () => {
    const drake = window.dragula(taskLists);
});

console.log(drake); // ReferenceError: drake is not defined

Для временной диагностики drake.containers логировался внутри обработчика удаления, где переменная доступна через лексическую область видимости.

Удаление DOM и очистка Dragula

На каждую колонку приходится один контейнер ul.task-list. При создании колонки ссылка на него отдельно добавляется во внутренний массив Dragula.

Register container 
drake.containers.push(taskList);

currentColumn.remove() удаляет колонку из DOM-дерева документа, но не изменяет массив drake.containers. Пока библиотека хранит ссылку, удаленный элемент остается достижимым в памяти. Такой элемент называют отсоединенным DOM-узлом.

Two independent states 
document DOM:       column removed
localStorage:       column removed
drake.containers:   old taskList reference remains

Поэтому перед удалением колонки ее список нужно удалить из массива библиотеки.

Container cleanup 
const currentTaskList = currentColumn.querySelector(".task-list");
const taskListIndex = drake.containers.indexOf(currentTaskList);

if (taskListIndex !== -1) {
    drake.containers.splice(taskListIndex, 1);
}

currentColumn.remove();
saveBoard();

indexOf() ищет тот же DOM-объект по ссылке. Валидные индексы массива начинаются с 0, а -1 означает, что элемент не найден.

Нельзя заменять проверку на if (taskListIndex): индекс 0 является falsy, хотя это найденный первый элемент, а -1 является truthy, хотя элемент не найден.

splice(taskListIndex, 1) изменяет исходный массив: первый аргумент задает позицию, второй - количество удаляемых элементов. Проверка на -1 обязательна, потому что splice(-1, 1) удалил бы последний рабочий контейнер.

Почему задачи не удаляются из drake.containers

В drake.containers зарегистрированы только зоны перетаскивания - элементы ul.task-list. Отдельные li.task туда не добавляются. Dragula читает текущих дочерних элементов зарегистрированного списка непосредственно из DOM.

Dragula model 
task-column
└─ task-list          // находится в drake.containers
   ├─ task            // обычный дочерний DOM-элемент
   ├─ task
   └─ task

При удалении одной задачи контейнер продолжает существовать, поэтому достаточно currentTask.remove(). При удалении колонки исчезает и сам контейнер, поэтому требуется очистить ссылку в Dragula.

Итог Step 2

  1. Состояние доски описывается массивом объектов, а не DOM-элементами.
  2. JSON.stringify() превращает данные в строку для хранения.
  3. JSON.parse() восстанавливает массив из строки.
  4. getItem() возвращает null, если ключ отсутствует.
  5. loadBoard() всегда возвращает массив, включая пустой [].
  6. Сохранение вызывается после изменения DOM, включая событие Dragula drop.
  7. Вложенный forEach() восстанавливает задачи внутри текущей колонки.
  8. Удаление элемента из документа не удаляет другие ссылки на него из памяти JavaScript.
  9. indexOf() и splice() очищают зарегистрированный контейнер Dragula.
