Step 1: Каркас, доска и создание колонки

На первом шаге собрали основу task manager: верхняя форма добавляет колонки, внутри доски есть горизонтальная область для колонок, а каждая колонка содержит форму для будущего добавления задач и отдельный список задач.

Главная идея этапа: доска - это не просто список карточек. У нее есть уровни ответственности: viewport для скролла, ряд колонок, отдельная колонка, форма задачи и список задач.

Структура интерфейса

Разметка строится вокруг нескольких смысловых уровней. Верхняя форма находится отдельно от scroll-области, чтобы не уезжать вместе с колонками.

DOM structure 
<section class="task-manager">
    <form class="add-column-form">...</form>

    <div class="task-board">
        <ul class="task-board__columns" id="columns">
            <li class="task-column">
                <header class="task-column__header">...</header>
                <form class="add-task-form">...</form>
                <ul class="task-list">...</ul>
            </li>
        </ul>
    </div>
</section>

БЭМ-нейминг

В проекте начали использовать БЭМ. Двойное подчеркивание показывает, что элемент является частью блока. Например, task-column__title - это title внутри блока task-column.

BEM pattern 
.task-column
.task-column__header
.task-column__title

.add-task-form
.add-task-form__input
.add-task-form__button

.task
.task__text
.task__delete-button

Важное правило: если есть класс вида block__element, должен быть понятен блок block. Поэтому add-task-form__input логичнее, чем add-task__input, если блока add-task в разметке нет.

Горизонтальная доска

Для колонок нужна отдельная область прокрутки. Важно не расширять всю страницу, а скроллить только внутреннюю доску.

Board viewport 
.task-board {
    width: 100%;
    overflow-x: auto;
    overflow-y: hidden;
}
  • overflow-x: auto включает горизонтальный скролл, когда колонок больше ширины контейнера.
  • overflow-y: hidden не дает board-обертке создавать лишний вертикальный скролл.
  • overflow-x: hidden здесь не подходит: он просто обрежет лишние колонки.

Ряд колонок

Сами колонки лежат внутри ul.task-board__columns. Для такого ряда удобнее flex, потому что новые колонки просто добавляются в конец линии.

Columns row 
.task-board__columns {
    display: flex;
    align-items: flex-start;
    gap: 16px;
    width: max-content;
}

width: max-content говорит ряду колонок: будь такой ширины, сколько нужно твоему содержимому. Если колонок много, ряд становится шире viewport, а .task-board дает горизонтальный скролл.

Ширина колонки

Колонка должна быть стабильной ширины. Для этого используется flex-сокращение:

Stable column width 
.task-column {
    flex: 0 0 280px;
}
  • Первый 0 - flex-grow: колонка не растет.
  • Второй 0 - flex-shrink: колонка не сжимается.
  • 280px - базовая ширина колонки.

Поэтому width: 100% на самой колонке здесь не нужен: он конфликтует по смыслу с задачей держать фиксированную ширину внутри flex-ряда.

Стандартный padding у ul

У ul в браузере по умолчанию есть левый отступ около 40px. Обнулить только list-style недостаточно: маркеры исчезнут, но padding останется.

List reset 
.task-board__columns,
.task-list {
    margin: 0;
    padding: 0;
    list-style: none;
}

Потом для .task-list можно отдельно вернуть нужный внутренний padding уже осознанно, например для отступов карточек внутри колонки.

Гибкий input и кнопка без переноса

У кнопки добавления колонки текст переносился на две строки. Это не проблема ширины формы, а поведение текста внутри кнопки.

No wrapping button text 
.add-column-form__button {
    white-space: nowrap;
}

white-space: nowrap запрещает перенос текста внутри кнопки. Input при этом может оставаться гибким и занимать свободное место.

DOMContentLoaded

JS запускается после события DOMContentLoaded, чтобы элементы из HTML уже существовали и их можно было найти через querySelector.

DOM ready 
document.addEventListener("DOMContentLoaded", () => {
    // Здесь уже можно искать элементы.
});

Проверка элементов

После поиска элементов идет проверка. Если какой-то важный элемент не найден, код останавливается через return.

Guard check 
if (!addColumnForm || !addColumnInput || !columnsItem) {
    console.log("Task manager elements not found");
    return;
}

return работает здесь, потому что код находится внутри функции обработчика DOMContentLoaded. На верхнем уровне файла return вызвал бы ошибку.

submit и preventDefault

Форма добавления колонки обрабатывается через событие submit. Это покрывает сразу два сценария: клик по кнопке submit и Enter внутри input.

Form submit 
addColumnForm.addEventListener("submit", (e) => {
    e.preventDefault();

    const columnTitle = addColumnInput.value.trim();
});

У формы есть стандартное поведение браузера: отправить форму и перезагрузить страницу. e.preventDefault() отменяет это стандартное действие. После этого форма остается на странице, а логикой управляет JS.

createElement и classList.add

Новая колонка создается через DOM API, а не через innerHTML. Это хорошая практика, потому что название колонки вводит пользователь.

Create DOM element 
const newColumn = document.createElement("li");
newColumn.classList.add("task-column");

document.createElement("li") создает новый DOM-элемент в памяти. Он еще не находится на странице. classList.add(...) добавляет CSS-класс этому элементу.

Свойства DOM-элементов

У созданного input можно настраивать свойства напрямую. Это не CSS-свойства, а свойства DOM-объекта, которые соответствуют HTML-атрибутам.

Input properties 
newColumnInput.type = "text";
newColumnInput.placeholder = "New task...";
newColumnInput.maxLength = 40;
  • type соответствует HTML-атрибуту type.
  • placeholder соответствует HTML-атрибуту placeholder.
  • maxLength соответствует HTML-атрибуту maxlength, но в JS пишется camelCase.

То есть эта JS-запись:

JS properties 
newColumnInput.type = "text";
newColumnInput.placeholder = "New task...";
newColumnInput.maxLength = 40;

примерно соответствует такой HTML-разметке:

HTML attributes 
<input type="text" placeholder="New task..." maxlength="40">

textContent

Пользовательский текст записывается через textContent. Это важно: textContent вставляет именно текст, а не HTML.

Safe text insertion 
newColumnTitle.textContent = columnTitle;

Если использовать innerHTML с пользовательским вводом, браузер будет воспринимать строку как HTML-разметку. Для портфолио и хорошей практики пользовательский текст лучше вставлять через textContent.

append

createElement создает отдельные элементы, но они еще не связаны между собой. append собирает эти элементы в DOM-дерево.

Append tree 
newColumnHeader.append(newColumnTitle);
newColumnForm.append(newColumnInput, newColumnButton);
newColumn.append(newColumnHeader, newColumnForm, taskList);
columnsItem.append(newColumn);

Сначала собираются маленькие части: заголовок получает title, форма получает input и button. Потом эти части добавляются внутрь колонки. Последний columnsItem.append(newColumn) вставляет готовую колонку в настоящий ul#columns на странице.

Mental model 
li.task-column
├─ header.task-column__header
│  └─ h2.task-column__title
├─ form.add-task-form
│  ├─ input.add-task-form__input
│  └─ button.add-task-form__button
└─ ul.task-list

Очистка input

После успешного создания колонки верхний input очищается:

Clear input 
addColumnInput.value = "";

Это setter-запись: свойству value input-элемента задается пустая строка. Пользователь видит, что форма готова принять следующее название колонки.

e.target.closest()

e.target - это конкретный элемент, с которого началось событие. Например, при клике по кнопке удаления target обычно будет сама кнопка. При submit формы target обычно будет форма.

Event target 
columnsItem.addEventListener("click", (e) => {
    console.log(e.target);
});

Но в динамическом интерфейсе нам обычно нужен не просто элемент, по которому кликнули, а ближайший смысловой контейнер: кнопка удаления, задача, колонка или форма. Для этого используется closest().

Closest button 
const deleteButton = e.target.closest(".task__delete-button");

Эта строка читается так: "начни с элемента, где произошло событие, и поднимайся вверх по DOM, пока не найдешь ближайший элемент с классом task__delete-button". Если такой элемент не найден, вернется null.

Важное уточнение: closest() ищет только вверх по DOM-дереву: сначала проверяет сам элемент, потом его родителя, потом родителя родителя и так далее. Он не ищет соседей и не ищет дочерние элементы.
Direction of closest 
button.task__delete-button
└─ closest(".task")
   └─ li.task

Для поиска вниз внутри уже найденного контейнера используется querySelector().

closest up, querySelector down 
const currentTask = deleteButton.closest(".task"); // вверх к li.task
const taskText = currentTask.querySelector(".task__text"); // вниз внутрь li.task
Guard after closest 
if (!deleteButton) {
    return;
}

Такая проверка нужна при делегировании событий. Обработчик висит на большом родителе columnsItem, поэтому клик может произойти где угодно внутри доски: по input, по заголовку, по карточке, по кнопке. Если клик был не по кнопке удаления, обработчик просто выходит через return.

Маршрут до текущей задачи

После того как кнопка удаления найдена, нужно удалить не кнопку, а всю задачу li.task. Поэтому от кнопки поднимаемся к ближайшей задаче.

Delete route 
const currentTask = deleteButton.closest(".task");
currentTask.remove();

Модель маршрута:

Click path 
click
└─ e.target
   └─ closest(".task__delete-button")
      └─ closest(".task")
         └─ remove()

closest при добавлении задачи

Та же идея используется при добавлении задачи. Нужно понять, из какой колонки пришел submit. Мы начинаем с формы задачи и поднимаемся к ближайшей колонке.

Current column route 
const taskForm = e.target.closest(".add-task-form");
const currentColumn = taskForm.closest(".task-column");
const taskList = currentColumn.querySelector(".task-list");

Так задача добавляется не в первую колонку и не во все колонки, а именно в список той колонки, где пользователь нажал Add.

Здесь маршрут смешанный: сначала два раза идем вверх через closest(), чтобы понять контекст, а потом идем вниз через querySelector(), чтобы найти нужный список внутри этой колонки.

closest() - это способ найти контекст действия. Кнопка говорит "удали мою задачу", форма говорит "добавь задачу в мою колонку".

Итог Step 1

  1. Разметка разделена на верхнюю форму, viewport доски, ряд колонок, колонку, форму задачи и список задач.
  2. Для горизонтальной доски используются overflow-x: auto, display: flex и width: max-content.
  3. Колонка держит стабильную ширину через flex: 0 0 280px.
  4. Стандартный левый padding у ul нужно обнулять отдельно.
  5. submit формы удобнее, чем отдельные обработчики click и Enter.
  6. preventDefault() отменяет стандартную отправку формы.
  7. createElement создает DOM-элемент в памяти.
  8. classList.add добавляет CSS-класс элементу.
  9. textContent безопасно вставляет пользовательский текст.
  10. append собирает элементы в дерево и вставляет готовую колонку в страницу.
  11. e.target.closest(...) помогает найти контекст события: форму, колонку, кнопку или задачу.
